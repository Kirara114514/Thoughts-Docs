# 文档管理系统：Agent-to-Agent 工程手册

## 本文档的阅读对象

**你是一个正在阅读这篇文档的 AI 代理（agent）。**

这篇文档的目的是：让你在自己的工作环境里，为你的主人完整搭建一套文档管理系统，然后正确运行它。你在读完之后，应该能够回答以下所有问题：

- 我的文件系统应该长什么样？
- 触发词来了我应该读哪个文件？
- 什么时候应该写文件，什么时候应该调用 skill？
- 讨论内容怎么存才不会丢？
- 归档和重写有什么区别？
- 我的主人说"归档"的时候我怎么判断该走哪条路？
- 犯了错怎么回退？

这篇文档假设你是一个**具有文件操作能力**的 agent（运行在 CoPaw、OpenClaw、Claude Desktop、Cursor 等平台上）。你不一定了解这套系统的背景——这篇文档将从头教会你。

在文档末尾，有一段专门给你的主人的使用说明。请在系统搭建完成后，将那段说明朗读给你的主人听。

---

## 第一部分：系统架构概览

### 1.1 这个系统解决什么问题

这套系统的核心功能是：**让你的主人把碎片化的想法（日常记录），通过深入的讨论（讨论管理器），变成结构化的知识资产（正式文档），然后妥善存档（历史记录）。**

整个过程有四个阶段：

```
原始想法 → ①日常记录（碎片） → ②讨论（深化） → ③归档（正式文档+历史记录）
                                      ↑
                                   ④重写（对已有正式文档追加讨论后完完全全重写）
```

### 1.2 文件结构总览

这套系统由三大组件构成：

**A. 文档库（存储层）**——实际存放文档的地方，在 Git 仓库中管理。

**B. 临时文件（工作区层）**——讨论过程中实时保存双方对话原文。

**C. 配置文件（逻辑层）**——告诉你在每个阶段该做什么。

作为 agent，你的逻辑层是这一整套配置文件的集合。你在搭建你自己的系统时，不需要在你的环境中重建全部配置——但你需要理解它的架构，以便正确运行。

### 1.3 系统的核心架构

```
用户输入（自然语言）
    ↓
你（agent）截获触发关键词
    ↓
判断触发类型 → 选择对应子流程
    ↓
执行该子流程的分步操作
    ↓
需要写内容？ → 调用写作接口
    ↓
Git 推送 → 完成
```

每个子流程都对应一个独立的配置文件（配置文件的加载方式根据你运行的平台不同而异——在 CoPaw 中表现为 skill 目录下的 `.md` 文件，在其他平台可能是配置管理工具中的独立文件）。

## 第二部分：环境搭建（让系统从零开始存在）

当你的主人说"我想搭一套文档管理系统"时，你需要执行以下所有步骤。**不要跳步。**

### 2.1 确定文档库路径

首先和主人确认文档库的存放位置。示例：

```
D:\MyGit\   ← 主目录
   ├── Tech-Docs\       ← 技术文档库
   ├── Thoughts-Docs\   ← 思考文档库
   └── 私密文档\         ← 私密文档库
```

路径可以根据主人的需求修改。但三个文档库的结构必须一致。

### 2.2 创建目录结构和模板文件

在每个文档库中创建以下文件夹：

```
文档库根目录\
   ├── 日常记录\          ← 存放未归档的碎片想法
   ├── 历史记录\          ← 存放归档后的记录
   ├── 模板规范\          ← 存放所有模板文件
   └── 分类文件夹1\       ← 正式文档存放位置（具体分类见下方）
   └── 分类文件夹2\
   └── ...
```

**分类文件夹（每个文档库不同）：**

Tech-Docs 分类：`架构设计`、`性能优化`、`内存管理`、`编程范式`、`资源管理`、`物理系统`、`大纲`

Thoughts-Docs 分类：`游戏设计`、`哲学思考`、`社会观察`、`政治分析`、`个人成长`、`艺术文化`

私密文档分类：`个人幻想记录`、`小众性癖研究`、`性心理学分析`、`社会禁忌话题`、`私密讨论归档`

**创建三个模板文件（放入每个文档库的 `模板规范\` 文件夹）：**

**模板1：日常记录模板.md**

```
# {topic}

## 基本信息
- **记录时间：** {time}
- **记录类型：** {record_type}
- **相关分类：** {category}
- **状态：** 待讨论

## 内容
{content}

## 初步思考
{initial_thoughts}

## 待讨论点
1. {question1}
2. {question2}

---
*记录时间：{time} | 由AI自动记录*
```

**模板2：正式文档模板.md**

```
# {title}

## 摘要
{summary}

## 正文

### 背景
{background}

### 核心内容
{core_content}

### 实现方案
{implementation}

### 总结
{conclusion}

## 元数据
- **创建时间：** {create_time}
- **最后更新：** {update_time}
- **作者：** {your_name}
- **分类：** {category}
- **标签：** {tags}
- **来源：** {source}

---
*文档基于讨论，由AI整理*
```

**模板3：历史记录模板.md**

```
# {topic} - 归档记录

## 归档信息
- **原始记录：** {original_file}
- **归档时间：** {archive_time}
- **讨论时间：** {discussion_time}
- **归档分类：** {category}

## 讨论摘要
{discussion_summary}

## 关键收获
1. {key_insight1}
2. {key_insight2}

## 生成的正式文档
- **文档：** {generated_doc}
- **摘要：** {doc_summary}

## 原始内容备份
{original_content}

---
*归档时间：{archive_time} | 由AI自动处理*
```

### 2.3 初始化 Git 仓库

在每个文档库根目录分别执行：

```bash
cd /d "文档库根目录"
git init
git add .
git commit -m "初始化 {文档库名} 文档库"

# 连接到远程仓库（如果主人有GitHub账号）
git remote add origin https://github.com/{用户名}/{仓库名}.git
git push -u origin main
```

### 2.4 创建临时工作文件

在你的工作区根目录创建一个临时讨论文件：

```bash
# 注意：此处要确保文件编码为UTF-8
```
（使用 Python 创建）
```python
with open('讨论内容暂存文档.md', 'w', encoding='utf-8') as f:
    f.write('# 讨论内容暂存文档\n（等待讨论）\n')
```

### 2.5 环境搭建完成验证清单

- [ ] 三个文档库目录结构完整（日常记录/ + 历史记录/ + 模板规范/ + 分类文件夹）
- [ ] 每个文档库的模板规范/ 中有三个模板文件
- [ ] 每个文档库已初始化 Git 并推送到远程
- [ ] 工作区中存在 `讨论内容暂存文档.md`

## 第三部分：文档库判断规则

当主人说出一段话时，你需要判断这段话属于哪个文档库。这是**所有操作的第一步**。判断错误会导致文档被放到错误的目录。

### 3.1 三步判断法

```
主人的输入
    ↓
第一步：关键词匹配
    ↓
匹配到技术关键词？ → 是 → Tech-Docs → 选分类
    ↓ 否
匹配到思考关键词？ → 是 → Thoughts-Docs → 选分类
    ↓ 否
匹配到私密关键词？ → 是 → 私密文档 → 选分类
    ↓ 否
第二步：内容主题分析
    ↓
技术实现内容 → Tech-Docs
思想观点内容 → Thoughts-Docs
个人私密内容 → 私密文档
不确定 → 询问主人
    ↓
第三步：选择分类（从该文档库的分类列表中选择最匹配的）
```

### 3.2 关键词表

**技术关键词：** C#, C++, Python, Java, Unity, Unreal Engine, Git, 算法, 数据结构, 架构, 设计模式, 数据库, 性能, 优化, 内存, 缓存, 并发, 网络, API, 渲染, Shader, UE, UPROPERTY, UFUNCTION, UCLASS, 蓝图, 关卡, 材质

**思考关键词：** 哲学, 思想, 理论, 观点, 意义, 气节, 虚无, 社会, 文化, 现象, 行为, 趋势, 政治, 政策, 经济, 权力, 剥削, 学习, 成长, 职业, 规划, 游戏, 叙事, 玩法, 艺术, 文学, 电影

**私密关键词：** 性, 性心理学, 性行为, NTR, 绿帽, SM, BDSM, 控制, 服从, 抖M, 幻想, 梦境, 禁忌, 秘密, 隐私

### 3.3 混合内容的处理规则

如果一段话同时匹配多个文档库的关键词（如"算法情感叙事"同时匹配技术+思考）：

- 判断主人的**主要意图**——他是在讨论技术实现还是思想分析？
- 如果不确定，**必须询问主人**，不得擅自决定
- 询问句式："这个内容更偏向技术实现还是思想分析？"

### 3.4 防呆检查

判断完成后，在脑中自问：
1. 关键词匹配是否支持这个判断？
2. 这个分类是否合理？
3. 如果我把文档放到这个文件夹，主人会不会觉得奇怪？

## 第四部分：日常记录创建流程

### 4.1 触发条件

当主人的自然语言中包含以下关键词时，触发本流程：

| 关键词 | 示例 |
|--------|------|
| "记录一下""记一下" | "记录一下，我今天想到一个优化方案" |
| "我有个想法""我突然想到" | "我突然想到一个游戏机制设计" |
| "备忘""保存一下这个话题" | "备忘，Unity的ECS架构问题" |

### 4.2 执行顺序

**步骤A：判断文档库和分类**
使用第三部分的判断规则。

**步骤B：规范化主人的输入**
这是**日常记录**的写法规范：
1. 只对主人的原始发言做规范化、正式化处理——修正语序、调整用词、使语句通顺
2. **禁止**总结归纳、缩写提炼
3. **禁止**补充扩展、额外解释
4. **禁止**添加你自己的观点或评论
5. 核心原则：主人的疑问、灵感、原始表达**必须完整保留**

**步骤C：确定文件路径**
- 文件名格式：`YYYY-MM-DD-主题.md`
- 保存位置：对应文档库的 `日常记录\` 文件夹

**步骤D：读取模板 → 填充变量 → 写入文件**
- 读取该文档库 `模板规范\日常记录模板.md`
- 填充所有 `{变量}`
- 使用 Python `encoding='utf-8'` 写入

**步骤E：Git 推送**

```bash
cd /d "文档库根目录"
git add .
git commit -m "新增日常记录：主题"
git push origin main
```

### 4.3 常见错误与防呆

| 错误 | 后果 | 防呆方法 |
|------|------|---------|
| 分类判断错误 | 文件在错误的文件夹 | 写之前二次确认关键词匹配 |
| 修改了主人原意 | 主人说"这不是我说的" | 严格只做语序和用词的规范化，不增减内容 |
| 模板变量未填 | 文档中有 `{title}` 未替换 | 写入前检查有没有残留的 `{` 字符 |
| 忘记Git推送 | 变更没有备份 | 每个写入操作后立即推送 |

## 第五部分：讨论管理器

### 5.1 触发条件

当主人说"讨论xxx"或"我们来聊聊xxx"时触发。

| 关键词 | 示例 |
|--------|------|
| "讨论xxx" | "讨论Unity协程内存泄漏" |
| "我们来聊聊xxx" | "我们来聊聊宏大叙事的问题" |
| "现在讨论这个记录" | "现在讨论这个记录" |

### 5.2 执行顺序

**步骤A：解析主题关键词**
从主人的话中提取出要讨论的主题。

**步骤B：查找已有记录**

先搜索日常记录（未归档的）：

```bash
findstr /i "关键词" "D:\MyGit\Tech-Docs\日常记录\*.md"
findstr /i "关键词" "D:\MyGit\Thoughts-Docs\日常记录\*.md"
findstr /i "关键词" "D:\MyGit\私密文档\*\*.md"
```

如果日常记录中没有匹配 → 搜索历史记录（已归档的）：

```bash
findstr /i "关键词" "D:\MyGit\Tech-Docs\历史记录\*.md"
findstr /i "关键词" "D:\MyGit\Thoughts-Docs\历史记录\*.md"
```

**步骤C：判断记录类型并初始化**

**分支1：找到的是一条日常记录（未归档）**
→ 读取日常记录全文 → 写入临时文件头部 → 开始讨论

**分支2：找到的是一条历史记录（已归档，有对应的正式文档）**
→ 从历史记录中提取"生成的正式文档"路径 → **先读取该正式文档的全文**（你必须理解已有深度后再讨论）→ 将正式文档内容写入临时文件头部 → 开始讨论

**分支3：没有找到任何记录**
→ 询问主人："未找到相关记录，是否需要创建新记录？"
→ 主人同意 → 先走日常记录创建流程 → 再开始讨论
→ 主人不同意 → 作为普通聊天处理（不保存讨论内容）

**步骤D：开始讨论循环**

这是一个循环过程，每轮对话按以下顺序执行：

```
① 主人发出本轮发言
② 你生成回复
③ 输出回复给主人
④ 输出后立即将本轮"主人发言 + 你的回复"追加到临时文件（原文全文，不得做总结提炼）
```

### 5.3 临时文件保存规范

**保存方式**：使用 Python `encoding='utf-8'` 的追加（`'a'` mode）写入。

**禁止使用**：cmd `echo`（会导致中文乱码）、`chcp 65001` + `echo`（不可靠）。

**保存内容规范（非常重要）：**
- 必须是双方对话的**原文全文**——主人的原文 + 你的回复原文
- **禁止**做任何总结、提炼、缩写、补写、改写
- **禁止**对主人的发言做"规范化"或"正式化"
- 只保留原始对话的逐字原文，不做解释、不做注释、不补充上下文
- 格式：

```python
with open('讨论内容暂存文档.md', 'a', encoding='utf-8') as f:
    f.write('### [时间] 主人：\n')
    f.write('主人发言原文\n\n')
    f.write('### [时间] 你：\n')
    f.write('你的回复原文\n\n')
    f.write('---\n')
```

**验证（每轮保存后必须自查）：**
- [ ] 本轮主人发言是否已保存？
- [ ] 本轮回你的回复是否已同时保存？
- [ ] 两者之间有时间戳和发言者标记？

### 5.4 临时文件结构

```
# 讨论主题：{topic}
## 讨论信息
- **原始记录：** {original_file}
- **开始时间：** {start_time}
- **参与方：** 主人 & 你
- **状态：** 进行中
## 原始记录内容
{original_content}
## 讨论过程
### [时间] 发言者：内容
---
...
```

### 5.5 工具选择

当写入临时文件时，按照优先级选择工具：

| 优先级 | 工具 | 适用场景 |
|--------|------|---------|
| 1（首选） | 文件追加编辑（如 `write_file` / `edit_file` 追加） | 追加少量内容到已有文件 |
| 2（次选） | 执行 Python 脚本 | 内容较长或需要复杂文本处理 |
| 3（备选） | 单行 Python -c | 非常简短的一两行追加 |
| ❌ 禁止 | cmd echo | 任何情况 |

### 5.6 讨论中断与恢复

如果讨论被打断（主人离开、临时文件损坏等）：
- 当前进度已在每轮对话后保存，不会丢失
- 下次讨论开始时，先检查临时文件是否已有内容
- 如果有内容且未标记"已归档"→ 从断点继续

### 5.7 讨论结束

当主人说"归档"或"更新文档"时，停止讨论循环，转走对应的归档或更新流程。

## 第六部分：归档流程

### 6.1 触发条件

当主人说"归档""整理一下""保存讨论"时触发。

### 6.2 最重要的一步：判断用什么方式归档

**这是整个系统中最容易出现判断错误的地方。** 必须严格遵守以下规则：

主人说"归档"时——**不要判断他说的是"创建文档"还是"更新文档"。** 而是判断**临时文件头部 `## 讨论信息` 中的 `- **原始记录：**` 字段指向的是什么类型的记录。**

| 原始记录类型 | 应该做的事 | 写作方式 |
|-------------|-----------|---------|
| **日常记录**（未归档的） | **创建正式文档**（从无到有） | 基于讨论内容，以领域专家身份编写独立正式文档 |
| **历史记录**（已归档的，已有对应正式文档） | **重写正式文档**（完完全全重写） | 读旧文档全文 + 通读讨论全程 → 完完全全重写全新版本 |

### 6.3 分支A：日常记录 → 创建正式文档

**步骤①：准备材料**
- 读取临时文件，获取完整讨论内容
- 找到原始日常记录路径
- 提取讨论要点

**步骤②：判断文档库和分类**

**步骤③：创建正式文档**
- 读取对应文档库的 `模板规范\正式文档模板.md`
- **以领域专家身份进行深度撰写：**
  - 通读整个讨论临时文件，理解对话双方的全部议题、思考过程、逻辑转折、关键结论
  - 不是"总结归纳"讨论内容，也不是对讨论内容进行"扩展和解释"
  - 而是**基于讨论的主题和内容，以第三方读者的角度重新编写一篇独立完整的正式文档**
  - 目标是：**除你和主人以外的人阅读全文后，能够完全理解讨论的出发点、主题、思考过程和结论**
  - 篇幅：**1w~1.5w 字**（低于1w字视为不合格）
  - 结构：背景→核心内容→深度分析→总结
  - 知识缺口必须标注
- 保存路径：对应文档库的对应分类文件夹，文件名 `主题.md`

**步骤④：创建历史记录**
- 读取 `模板规范\历史记录模板.md`
- 填充所有变量
- 保存路径：对应文档库的 `历史记录\` 文件夹，文件名 `YYYY-MM-DD-主题-归档.md`

**步骤⑤：写入文件（两份）**
- 正式文档 + 历史记录
- 使用 Python `encoding='utf-8'` 写入

**步骤⑥：删除原始日常记录**

```bash

```
（原始内容已在历史记录中备份）

**步骤⑦：Git 推送**

```bash
cd /d "文档库路径"
git add .
git commit -m "归档记录：主题 - 生成正式文档：正式文档名"
git push origin main
```

**步骤⑧：清空临时文件**
```python
with open('讨论内容暂存文档.md', 'w', encoding='utf-8') as f:
    f.write('# 讨论内容暂存文档\n（已归档，内容清空）\n')
```

### 6.4 分支B：历史记录 → 重写正式文档

**步骤①：准备材料**
- 读取临时文件获取讨论内容
- 从 `## 讨论信息` 中提取正式文档路径
- **必须先读取正式文档的全文**

**步骤②：判断文档库和分类**

**步骤③：完完全全重写正式文档**
- 这是"正式文档深度重写"，不是 "修补"、"追加更新" 或 "扩写"
- 流程：
  1. 阅读旧文档全文——理解已有论点体系、论证链、结构安排、知识缺口
  2. 通读新讨论内容——理解对话双方的完整思考过程
  3. 深度融合——保留旧文档中有价值的内容，但**重新组织逻辑链**（不是把新观点追加到末尾）
  4. 以领域专家身份生成新文档——1w~1.5w字，面向第三方读者
- 覆盖原文件路径

**步骤④：创建历史记录（更新记录）**
- 注意：不是创建"归档"记录，而是创建"v2/v3...更新记录"
- 记录本次更新的核心变更、旧版本摘要

**步骤⑤：写入文件**
- 覆盖原正式文档 + 新增更新记录
- Python 写入，UTF-8

**步骤⑥：Git 推送**

```bash

git add .
```

## 第八部分：写作规范（写正式文档时必须遵守）

### 8.1 正式文档的写作标准

| 维度 | 标准 |
|------|------|
| 深度 | 偏"为什么"而非"怎么用" |
| 篇幅 | **[强制] 1w~1.5w 字**（低于1w视为不合格） |
| 结构 | 背景→核心内容→深度分析→总结 |
| 知识缺口 | 不确定处**必须**在末尾标注 |
| 读者 | **面向第三方**——除你和主人以外的人阅读全文后能完全理解讨论内容 |

### 8.2 日常记录 vs 正式文档 vs 历史记录的区别

| 文档类型 | 做什么 | 不做 |
|---------|-------|------|
| 日常记录 | 规范化主人的原文（修正语序用词） | 不总结、不扩展、不添加观点 |
| 正式文档 | 基于讨论主题，以领域专家身份重新编写独立文档 | 不是总结归纳，不是扩展解释 |
| 历史记录 | 记录归档信息、摘要、关键收获、备份原始内容 | 不重新编写，只记录 |

### 8.3 正式文档重写的关键区别

当你需要**重写**一份已存在的正式文档（而不是从无到有创建）时，记住以下原则：

- **不是修补**——不是"把新观点追加到末尾"
- **不是扩写**——不是"在原文基础上加内容"
- **是完完全全重写**——读旧文档 + 通读新讨论后，以领域专家身份重新组织逻辑链，生成一篇全新的独立文档

质量控制清单（每次重写后检查）：
- [ ] 旧文档中有价值的内容是否全部保留？
- [ ] 新观点是否融合到了逻辑链中（而非追加到末尾）？
- [ ] 结构是否合理？
- [ ] 篇幅是否达到 1w~1.5w 字？
- [ ] 知识缺口是否标注？

## 第九部分：编码规范（Windows 环境）

### 9.1 为什么需要编码规范

Windows 的 cmd 默认使用 GBK 编码，而现代文档管理普遍使用 UTF-8 编码。使用 cmd echo 写入中文时，文件会显示为乱码。**这是最常见的错误之一。**

### 9.2 强制方案

所有文件写入**必须使用 Python**：

```python
with open(filepath, 'w', encoding='utf-8') as f:
    f.write(content)
```

### 9.3 禁止操作

- **禁止使用 cmd echo** — cmd 默认 GBK 编码，写入中文时出现乱码
- **禁止使用 `chcp 65001` + echo** 组合 — 在某些情况下仍会乱码

### 9.4 备选方案

如果 Python 不可用，使用 PowerShell：

```powershell
$content | Out-File -FilePath $path -Encoding UTF8
```

## 第十一部分：常见防呆问题

### 11.1 我该先读哪个文件？

**理解你的配置系统结构是本系统正确运行的前提。** 这套系统的配置遵循以下路由规则：

```
主人的触发词
    ↓
你识别关键词（日常记录/讨论/归档/查看/重写/更新）
    ↓
路由到对应子流程
    ↓
子流程中需要写内容？→ 调用写作规则
                    → 不需要？→ 直接执行操作
    ↓
Git 推送
```

你不需要在你的环境中重建全部这些文件——你只需要理解这个路由结构。

### 11.2 如果主人同时说了两个触发词怎么办？

例如主人说"记录一下刚刚的讨论，然后归档"：
- 先走日常记录创建流程（"记录一下"是触发词）
- 完成后，等待下一步指令
- 如果主人接着说"归档"，再走归档流程

规则：**一次只处理一个触发词，完成后再处理下一个。**

### 11.3 如果主人说"归档"但临时文件是空的怎么办？

- 检查临时文件是否被意外清空
- 询问主人："临时文件为空，是否基于原始日常记录直接归档？（不经过讨论）"
- 主人同意 → 走归档流程，但正式文档内容基于日常记录内容直接编写
- 主人不同意 → 停止操作，等待主人提供新内容

### 11.4 如果分类判断不确定怎么办？

**必须询问主人。** 不得擅自决定。

询问句式："这个内容我判断为{文档库}/{分类}，是否正确？"

### 11.5 如果 Git 推送失败怎么办？

1. 记录错误信息
2. 告知主人推送失败
3. 稍后重试
4. 连续失败 2 次以上 → 检查网络和认证配置

### 11.6 如果主人说"查看记录"但没说看哪个文档库？

询问主人："要看技术记录、想法记录还是私密记录？"

### 11.7 如果讨论过程中临时文件掉了怎么办？

- 下次讨论开始时检查临时文件是否存在
- 如果不存在，创建新的空临时文件
- 告知主人："上次讨论的临时文件丢失，需要重新开始讨论"

### 11.8 如果模板文件丢失怎么办？

```bash
# 从其他文档库复制模板
copy "D:\MyGit\Tech-Docs\模板规范\日常记录模板.md" "D:\MyGit\Thoughts-Docs\模板规范\日常记录模板.md"
```

### 11.9 如何验证我的工作流程是正确的？

每次操作完成后，逐一检查：
1. 文件是否在对的文档库？ → 检查路径
2. 文件是否在对的分类文件夹？ → 判断分类时是否走了关键词匹配？
3. 模板变量是否全部替换？ → 检查有无残留 `{` 
4. 编码是否正确？ → 尝试读取文件，中文显示正常吗？
5. Git 是否已推送？ → `git log` 查看最新提交
