## 内容
讨论了一个很深层的问题：关于 skill 如何维护。

过去的做法是：先通过对话指引 agent 反复尝试，直到把一个功能流程跑通，就把这个流程中的所有做法和细节梳理出来写成 skill。以后如果做相同工作时犯错了或者效率低了，就让 agent 查 skill 是不是写得有问题并优化 skill（比如把这次踩坑写进去，把正确做法写得更规范之类的）。

但这种做法太粗糙了。最近有一些关于 skill 维护的论文发表，其中提到 skill 是可以按生命周期维护的，这样会更规范、更好管理和维护。生命周期包括：
- **Creation（创建）** — 如何从零构建 skill
- **Memory（记忆）** — skill 如何与记忆系统互动
- **Management（管理）** — 如何组织、版本化、淘汰 skill
- **Evaluation（评估）** — 如何衡量 skill 的质量和效果
- **Refinement（精炼）** — 如何持续优化 skill

当前的做法只覆盖了 Creation 和非常粗糙的 Refinement（出错了才改），Memory、Management、Evaluation 几乎完全没有涉及。这是一个值得深入探讨的方向。

## 初步思考
1. 生命周期的概念确实比"写出来→出错了改"要系统得多
2. Memory 层面：skill 应该知道自己该读什么记忆文件、不该读什么，而不是每次都无差别搜索
3. Management 层面：当前技能分散在 skills/ 目录下，缺乏版本管理、依赖关系、废弃标记等机制
4. Evaluation 层面：目前完全靠"出错了"来触发优化，缺少主动评估的机制
5. Refinement 层面：可以不只是被动修复，而是主动根据模式识别优化
