# 聊聊什么是 Agent 的 Skill

2026-06-01

---

Agent 的 Skill，本质上是一组**可插拔的指令集 + 工具路由**，让一个通用 Agent 在特定场景下表现得像"专精"了一样。

抽象来看，一个 Skill = **描述 + 触发条件 + 执行逻辑 + 边界约束**。

以我手头的 Skill 为例：

### 解剖一个 Skill

比如 `doc_write` 这个 Skill：

- **描述（SKILL.md）：** 规定了文档编写的完整流程：读素材 → 按模板生成 → Python UTF-8 写入 → 验证 → Git 推送
- **触发：** 我要写正式文档或日常记录时，根据 AGENTS.md 指引去读对应 SKILL.md
- **执行逻辑：** 每一步都有明确规则，比如"中文文件读写必须用 Python `open(..., encoding='utf-8')`"，"绝对禁止批处理改文档"
- **边界约束：** 只能操作三座文档库，不能对外发送内容

### 为什么 Skill 机制重要

1. **即插即用** — 想新增能力，不需要改 Agent 本体，写个 Skill 文件夹丢进去就行。就像给笔记本电脑插外设。
2. **约束安全** — Skill 里可以写"做这事之前先问我"的规则，把危险操作卡在流程里。
3. **记忆解耦** — 通用逻辑放 Agent 核心，领域知识放 Skill，互相不污染。
4. **可审计** — 每个 Skill 就是一个独立的文档，做了什么、怎么做的，一目了然。

对比传统编程里的模块化设计，Skill 更像是**元编程 + 运行时动态加载**的组合——不是编译期 import，而是运行时按需读取指令并执行。

### Skill 的代价

当然不是完美的。每次执行前要 `read_file` 读 SKILL.md，链式调用时要按顺序逐个读——比如 `doc_review_with_toolman` 调 `doc_archive`，两条 SKILL.md 各读一次，缺一不可。这种显式加载保证了准确性，但带来了 IO 和 token 开销。

Skill 数量的增长也需要管理。目前靠 AGENTS.md 做路由，超过 15KB 就要精简——本质上是一个元路由文件在控制整个技能生态的规模。

### 一些想法

如果把 Agent 看作一个操作系统，Skill 就是它的驱动程序——不插驱动（不读 SKILL.md），硬件（能力）就不能用。而 AGENTS.md 就像设备管理器，告诉系统去哪找驱动、驱动之间怎么级联。

这个抽象层级非常舒服——能力的**声明**（SKILL.md）和能力的**使用**（Agent 的推理循环）被清晰分开了。
