# 日常记录：Agent引入后的效率恐慌与思考负担

> 日期：2026-06-02
> 状态：待归档

## 事件

今天聊到了一个有趣的话题——在工作中引入 AI Agent（如 Claude Code、Open Code 等）之后，不同体量的公司、不同岗位的人感受截然不同。

## 核心内容

### 大厂视角：效率恐怖到恐惧

按常理排期两周的需求单，考虑到有 Agent 加速，压缩到一周零一天。结果实际上第二天就完成了，而这两天里几乎没怎么"工作"——完全在看文档学习或摸鱼，只是在间隙中看看 Agent 怎么样了，给予指引。

在这种恐怖的效率和突然变轻松的工作环境里，感受到的不是喜悦，而是恐惧和焦虑。因为这一定不是常态——工作过于轻松，公司会很快开始裁员，直到留下的人只能勉强刚好扛住工作量的程度。

所在项目组大约两千人，这是身处其中的真实感受。

### 中厂视角：思考负担变重了

一个在中厂做微信小程序小游戏的朋友反馈，他觉得引入 Agent 后，"思考负担"变重了。

这可能和公司架构有关：
- 一人兼任多职（客户端资深老师傅 + TA 等）
- 朝令夕改、快速迭代
- 文档体系不完善

即便使用 Agent，也有很重的负担——因为 Agent 能省的是"执行时间"，但不能省"决策和协调时间"。

### 底层差异分析

两种体验的核心分歧不在于 Agent 本身，而在于**所在环境的数字化程度和文档完善度**：

| 维度 | 大厂 | 中厂 |
|------|------|------|
| 文档体系 | 完善，喂给 AI 就能覆盖大量问题 | 薄弱，Agent 缺乏上下文 |
| 需求迭代 | 慢，转型周期长 | 快，朝令夕改 |
| 角色定位 | 专注，分工明确 | 多职兼任，职责模糊 |
| Agent 效用 | 替代大量执行工作 | 只替代部分执行，决策负担还在 |

大厂文档完善，喂给 AI 就能一劳永逸解决很多问题，剩下需要人处理的降低不少。而中厂文档体系薄弱，Agent 缺乏上下文，用起来反而增加负担。

### 恐惧的根源

这种恐慌来自一个推演：当 Agent 让效率提高到某个阈值，公司开始裁人。裁员后剩下的人刚好扛住被 Agent 提效后的工作量。然后 Agent 继续迭代、继续提效——继续裁员。直到剩下的人忙到极限，或者需求复杂度上升到 Agent 也搞不定的程度。

一个猜想：大厂项目转型慢，目前接到的需求大概还是"上个时代"的难度。再过半年左右，需求可能就会复杂到（或者数量多到），用 Agent 也必须每天加班的程度——或者裁员裁到了那个程度。
